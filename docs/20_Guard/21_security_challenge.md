# Security課題

## この文書について

この文書は、AIネイティブ開発基盤における **Security課題** を整理するものです。

ここでは、失敗事例や事故分析の詳細には踏み込みません。  
失敗パターン、事故例、ヒヤリハット、関連URLの整理は、別途付録として扱います。

---

## 1. Security課題とは

Security課題とは、AIを開発作業に参加させるときに、情報・権限・操作・外部接続をどのように守るかという課題です。

従来の開発でも、機密情報の取り扱い、権限管理、secret管理、静的解析、監査ログ、CI/CDでの検査などは重要でした。

withAI開発では、そこに次の観点が加わります。

```text
AIに何を読ませてよいか
AIに何を実行させてよいか
AIにどの外部toolへ接続させてよいか
AIが生成したものをどう検査するか
人間がどこで承認・停止・確認するか
```

つまりSecurity課題は、単に「AIに機密情報を入力しない」という話に留まりません。

AIが開発環境の中で作業する場合、AIはファイルを読み、コードを書き、コマンドを実行し、外部APIや開発toolに接続する可能性があります。

そのため、この研究では、AIを安全に使うには、AIの作業範囲と権限の境界を明確にする必要があるのではないか、と見ています。

---

## 2. なぜ重要か

LLMが普及し始めた頃から、機密情報をAIに入力してよいのかという問題は指摘されていました。

例えば、社内コード、会議メモ、顧客情報、認証情報などをAIに入力すると、情報管理上のリスクが発生します。

withAI開発では、この問題がより具体的になります。

LLMを単なる相談相手として使うだけであれば、人間が入力した内容が主なリスクになります。

一方で、AI agentやAI coding toolを開発環境で使う場合、AI自身が次のような作業を行う可能性があります。

```text
repository内のfileを読む
設定ファイルを読む
.envやcredentialを探す
コードを編集する
テストやbuild commandを実行する
外部APIへrequestする
MCPなどを通じて外部toolに接続する
PRやCI/CDに関与する
```

この研究では、AIが開発作業に参加するほど、Security課題は「情報を入力してよいか」から「AIに何を触らせてよいか」へ広がると見ています。

---

## 3. AI利用時に発生する主なリスク

Security課題では、主に次のリスクを見ます。

### 3.1 情報漏洩

AIが機密情報、API key、token、顧客情報、社内資料などを読んだり、外部へ送信したりするリスクです。

従来は、人間が誤ってsecretをcommitしたり、外部サービスに貼り付けたりすることが問題でした。

withAI開発では、AIが作業中に機密情報へ到達する可能性があります。

```text
AIが.envを読む
AIがconfigからtokenを見つける
AIがログやテストデータから機密情報を拾う
AIがPR本文やコメントに値を出力する
```

そのため、secretを検知するだけでなく、AIから見えない場所に置く、AIが読めない権限にする、AIが使っても安全なscopeにする、といった設計が重要ではないか、と考えています。

### 3.2 権限の過剰付与

AIに強すぎる権限を与えると、意図しない操作が大きな影響を持つ可能性があります。

例えば、AIに本番DBへ接続できるtoken、cloud resourceを削除できる権限、広範囲のrepository access、外部serviceへの書き込み権限を与えると、操作ミスや誤判断がそのまま事故につながります。

この研究では、AIに与える権限を作業内容に応じて分けることを基本方針候補として見ています。

```text
read
write
execute
network
deploy
production access
external tool access
```

### 3.3 危険な操作

AIがコマンドを実行できる場合、file削除、DB更新、cloud resource操作、外部API実行などが問題になります。

```text
危険なshell command
project外のfile変更
本番環境への接続
backup削除
外部APIへの副作用あるrequest
CI/CD設定変更
```

このような操作に対しては、実行前に人間承認を挟む、sandbox内に閉じる、危険コマンドをblockする、外部副作用を持つ操作を制限する、といった対応が有効ではないか、と見ています。

### 3.4 外部接続

AI agentは、MCPや各種connectorを通じて、GitHub、Slack、Jira、Google Drive、社内tool、cloud環境などに接続する可能性があります。

これは便利である一方、Security境界を広げます。

```text
どのtoolに接続してよいか
どのdata sourceを読んでよいか
どの操作を実行してよいか
OAuthやtokenのscopeは適切か
外部入力によるprompt injectionを受けても危険操作をしないか
```

AIにとって外部入力は、単なる参考情報ではなく、AIの行動に影響する可能性があります。

そのため、AIが読む情報と、AIが実行できる操作を分けて考えることが重要ではないか、と考えています。

### 3.5 生成物リスク

AIが生成したコード、設定、CI/CD、Dockerfile、依存関係、ドキュメントにもSecurityリスクがあります。

```text
hardcoded secret
入力検証漏れ
認可漏れ
危険な依存追加
存在しないpackage名
危険なCI/CD設定
過剰な権限を持つ設定
```

そのため、この研究ではAI生成物を人間が読むだけでなく、機械的な検査やCI gateに通すことを重要な対応候補として見ています。

## 4. Security課題の主な観点

Security課題は、単一の対策ではなく、複数の観点を組み合わせて考えると整理しやすいです。

### 4.1 情報管理

AIに読ませてよい情報と、読ませてはいけない情報を分けます。

最も安全なのは、AIが触れる環境に機密情報を置かないことです。

ただし、API連携のテストや実環境に近い検証では、AIの手の届く場所に一定のcredentialや設定が必要になる場合があります。

その場合は、scopeを絞る、短時間で失効させる、検証専用のcredentialを使う、操作対象をsandboxに限定するなどの工夫が必要になります。

### 4.2 権限管理

AIにどの操作を許可するかを分けます。

```text
読むだけ
file編集まで
test実行まで
外部通信まで
deployまで
本番操作まで
```

このように、作業内容ごとに権限を段階化することが有効ではないか、と見ています。

### 4.3 実行制御

AIがcommandを実行する場合、危険操作をどう止めるかが重要です。

```text
危険commandのblock
実行前承認
sandbox環境
project外への書き込み制限
network制御
本番環境への接続禁止
```

などが考えられます。

### 4.4 生成物検査

AIが生成したものは、人間レビューだけでなく、機械的な検査にも通すことが有効ではないか、と見ています。

```text
secret scanning
SAST
dependency scanning
license check
CI test
policy as code
IaC scan
container scan
```

これは、人間レビューを不要にするという意味ではありません。

むしろ、人間が判断すべき点に集中するために、機械的に止められるものは先に止めるという考え方です。

### 4.5 監査・検知

AIが何を読んだか、何を実行したか、どのfileを変更したか、どのtoolに接続したかを追えることも重要です。

```text
操作ログ
承認ログ
command実行履歴
file変更履歴
外部API call履歴
token利用履歴
CI/CD実行履歴
```

問題が起きた後に原因を追うには、このような記録が必要になる可能性があります。

### 4.6 人間承認

Security課題では、人間がどこでOKを出すかも重要です。

ただし、単に承認画面を出せば十分ではありません。

人間が判断できる情報が揃っている必要があります。

```text
何を実行しようとしているか
どのfileに影響するか
外部副作用があるか
どの権限を使うか
rollback可能か
本番環境に影響するか
```

この研究では、「承認を挟む」だけでなく、「承認できる状態を作る」ことが重要ではないか、と見ています。

## 5. Security課題を層で見る

Security課題は、多層で考えると整理しやすくなります。

```text
Layer 0: そもそもAIから見えない場所に置く
Layer 1: AIに読ませる情報を制限する
Layer 2: AIのread / write / execute / network権限を分ける
Layer 3: AIの生成物をcommit / PR / CI前に検査する
Layer 4: 漏洩や危険操作を検知する
Layer 5: 漏れた後に失効・追跡・復旧する
Layer 6: 人間がOKを出すための教育・policy・承認設計を整える
```

重要なのは、後段の検知だけに頼らないことです。

```text
見せない
制限する
実行前に止める
生成物を検査する
漏洩を検知する
復旧できるようにする
人間が判断できるようにする
```

という複数の層を組み合わせる必要があるのではないか、と考えています。

## 6. 人間側のリテラシーと運用

withAI開発では、人間側のリテラシーも重要になります。

AIを使う人が、次を理解していないと、安全な運用は難しくなります。

```text
AIに貼ってはいけない情報
AIに読ませてはいけないfile
AIに許可してはいけないcommand
AIに渡してはいけないtoken
本番環境に接続してはいけない場面
外部tool接続時に確認すべきscope
生成物をそのまま信用してはいけない理由
```

従来のネットリテラシーやSecurityリテラシーは、やや曖昧に扱われることもありました。

しかし、withAI開発では、AIに与える情報や権限がそのまま操作につながるため、曖昧な注意では不十分です。

組織でAIを使う場合は、次を明確にすることが有効ではないか、と見ています。

```text
AI利用時の禁止事項
AIに渡してよい情報
AIに許可してよい操作
AI利用時のreview基準
AIが生成したcodeの検査手順
credentialの扱い
外部接続の承認基準
事故時の対応手順
```

これは、企業がwithAI開発へ進むうえで、早い段階で整備すべき領域だと考えられます。

---

## 7. 現時点で見えていること

現時点では、Security課題は次のように整理できます。

```text
withAI開発におけるSecurityは、
AIを禁止することではなく、
AIが安全に作業できる範囲を設計することである。
```

Security課題で重要なのは、AIの能力を単純に増やすことではありません。

この研究では、AIに何を許可し、何を禁止し、どこで人間が確認し、どこで機械的に止めるかを設計することが重要だと見ています。

特に重要な候補は、次の点です。

```text
AIに見せる情報を制御する
AIに与える権限を絞る
AIの実行操作を制限する
AI生成物を検査する
AIの操作ログを残す
人間が判断できる形で承認する
```

AIが開発作業に深く関わるほど、Securityは後付けではなく、開発基盤そのものに組み込むべき要素として扱う必要があるのではないか、と見ています。

---

## 8. 今後扱う付録

失敗事例、事故分析、ヒヤリハット、Prompt Injection、MCP、Supply Chain、Policy / Governance などの詳細は、別途付録として扱います。

付録では、次の観点で整理する予定です。

```text
どのような失敗が起きたか
AIはどこに関与したか
人間はどこで止められたか
どのGuardがあれば防げたか
既存Securityと何が違うか
```

この付録を通して、Security課題の見立てを再確認します。
