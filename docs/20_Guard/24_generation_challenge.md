# withAI開発で気をつけたい「生成物」と「正本」の話

## これは何の話か

LLMやAI Agentを使って開発していると、Agentが目の前のエラーや差分を見て、すぐに直せそうなファイルを編集しようとすることがあります。

それ自体は便利ですが、開発現場には「直接直してよいファイル」と「別のファイルから作られるファイル」があります。

この区別を間違えると、いったん動いたように見えても、次に生成し直したときに変更が消えたり、仕様と実装がずれたりします。

この課題では、そうした **生成物と正本の取り違え** を扱います。

```text
正本
↓
生成する手順
↓
生成物
↓
検証結果 / 配布物
```

たとえば、生成されたコードを直接直すのではなく、元になっている仕様やschemaを直すべき場面があります。

---

## 身近な例

### OpenAPI Generator

OpenAPI Generatorでは、OpenAPI SpecからAPI client、server stub、documentationなどを生成します。

この場合、生成されたclient codeやserver stubを直接直すより、元のOpenAPI Specやtemplateを直して再生成する方が自然な場面があります。

- 正本: OpenAPI Spec
- 生成物: client SDK、server stub、docs など
- 気をつけること: 生成物だけを直すと、次回生成で変更が消える可能性がある

### Buf

Bufは、Protocol Buffersを扱うためのtoolchainです。schemaのbuild、lint、format、breaking change check、code generationなどを扱います。

この場合も、生成されたコードだけを見るのではなく、元のProtobuf schemaや生成手順を見る必要があります。

- 正本: Protobuf schema / module
- 生成物: generated code、generated SDK、docs など
- 気をつけること: schemaと生成物の関係を崩さない

### Protocol Buffers

Protocol Buffersでは、`.proto` ファイルを元に各言語のコードを生成します。

生成されたJava、TypeScript、Goなどのコードを直接直すのではなく、必要なら `.proto` を直して、再生成するのが基本になります。

- 正本: `.proto` file
- 生成物: 各言語のgenerated code
- 気をつけること: 生成物の修正ではなく、元の定義を直すべき場面がある

---

## なぜAI Agentで問題になりやすいのか

人間でも、生成物を直接直してしまうことはあります。

ただ、AI Agentは特に、目の前のエラーや差分に強く反応しがちです。

```text
このファイルを直せばテストが通りそう
この差分だけ直せばよさそう
前回もここを直したから今回もここでよさそう
```

でも、そのファイルが生成物だった場合、本当に直すべき場所は別にあるかもしれません。

```text
OpenAPI Spec
.proto file
schema
template
generator config
```

この研究では、AI Agentに任せるほど、「今見えているファイル」ではなく「どの段階を直すべきか」を明示することが重要ではないか、と見ています。

---

## withAI開発で見ておきたいこと

AI Agentに修正を任せる前に、少なくとも次の情報がProject側にあると安全です。

```text
どれが正本か
どれが生成物か
生成コマンドは何か
検証コマンドは何か
生成物を直接編集してよい例外はあるか
再生成後に何を確認するか
```

これは、AI Agentだけのためではありません。

人間が参加するときにも、同じ情報があると迷いにくくなります。

---

## 短命Agentとの関係

AI Agentを作業ごとにリセットする運用では、Agentの記憶に頼らず、Project側の情報を毎回読み直すことになります。

このとき、Project側に「正本・生成手順・生成物・検証手順」が明示されていれば、毎回新しいAgentでも同じ前提で作業しやすくなる可能性があります。

逆に、そこが書かれていないと、Agentは目の前のファイルだけを見て判断しやすくなります。

つまり、この研究では短命Agentをうまく使うには、Project側に次のような情報を残すことが重要ではないか、と考えています。

```text
このファイルは生成物
元はこのschema
修正はこの正本へ
再生成はこのコマンド
確認はこのテスト
```

---

## この課題で伝えたいこと

生成物課題は、「AIが触ってはいけないファイルを触る」というだけの話ではありません。

この研究では、より正確には **AI Agentが変更すべき段階を間違えないようにする話** として整理しています。

```text
後段の生成物を直すのか
前段の正本を直すのか
生成手順を直すのか
検証手順を直すのか
```

withAI開発では、Agentに作業を任せる前に、この前後関係をProject側に書いておくことが重要ではないか、と見ています。

AI Agentは速く作業できますが、速く間違えることもあります。

だからこそ、この研究ではProject側に「何が正本で、何が生成物か」を明示しておくことを、withAI開発における基本的なGuard候補として見ています。
