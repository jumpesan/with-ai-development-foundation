# AI時代のSpec設計の整理

## この資料について

この資料は、withAI開発における **Spec（仕様）設計** の概要を共有するためのものです。

ここでいうSpecとは、AI Agentに実装させる前に、今回の作業の目的・範囲・制約・完了条件を明確にする情報です。

従来の設計書や要件定義書と近い部分はありますが、withAI開発では少し役割が変わります。

```text
従来:
人間同士が合意し、分業し、製造・試験するための工程成果物

withAI:
AIに渡す作業範囲、守る制約、検査する基準を明確にするための境界情報
```

重要なのは、Specを重い設計書として復活させることではありません。

AIが速くズレないように、作業前に必要な境界を定義することです。

---

## Specがないと何が起きるか

AIは、曖昧な依頼でも何かしら形にできます。

たとえば、次のような依頼でも実装自体は進められます。

```text
ログイン画面をいい感じに直して
検索機能を追加して
エラー処理を整理して
既存に合わせて実装して
```

ただし、これは危険です。

```text
速く作る
=
正しく作る
ではない
```

Specがないと、AIは不足している前提を自分で補います。

その結果、次のような問題が起きやすくなります。

```text
本来の目的と違う変更になる
変更範囲が広がる
やらないはずの設計変更が混ざる
完了条件が曖昧になる
検証方法が後付けになる
人間がレビューしにくい差分になる
```

Spec設計は、AIに実装させる前に「今回何をするのか」と「何をしないのか」を揃えるための設計です。

---

## ContextとSpecの違い

ContextとSpecは似ていますが、役割が違います。

```text
Context:
このProjectで常に守るルール

Spec:
今回の作業で何を達成するか

Issue / PR:
その作業を実行・検査する単位
```

例:

```text
Context:
このrepoではpnpmを使う。
generated filesは直接編集しない。
認証処理を変更する場合は関連guideを読む。

Spec:
ログイン画面の入力バリデーションを追加する。
認証方式は変更しない。
空入力・不正メール形式のエラーを追加する。

Issue / PR:
このSpecを実装するための作業チケットと変更提案。
```

Contextなしでは、AIは現場ルールを知りません。

Specなしでは、AIは今回の目的・範囲・完了条件を知りません。

---

## Specに書くとよい情報

Specには、AIが今回の作業を正しく絞り込むための情報を書きます。

```text
目的
対象ユーザー
変更範囲
変更しない範囲
守る制約
入力と出力
完了条件
検証方法
リスク
未決事項
```

特に重要なのは、やることだけでなく、やらないことを書くことです。

```text
認証方式は変更しない
DB schemaは変更しない
UI文言だけを変更し、APIは変更しない
既存の生成物は直接編集しない
```

AIは作業を広げやすいため、Out of Scopeを明示するとズレを減らしやすくなります。

---

## Specは自然文だけでなく契約にもできる

Specは自然文だけとは限りません。

API、イベント、データ構造、外部連携などでは、機械可読な仕様が役に立ちます。

```text
OpenAPI
AsyncAPI
JSON Schema
Protocol Buffers
DB schema
state machine
```

このような形式は、AIが勝手にinterfaceを解釈する余地を減らします。

たとえばAPIを作る場合、自然文だけで「検索APIを追加して」と書くよりも、次の情報がある方が安定します。

```text
endpoint
method
request schema
response schema
error pattern
authentication
backward compatibility
```

自然文のSpecと、機械可読なContract Specを組み合わせることで、AIに任せる範囲を絞りやすくなります。

---

## Specは段階に分ける

AIにいきなり実装させるのではなく、段階に分けるとズレを減らしやすくなります。

```text
目的を決める
↓
仕様を書く
↓
実装方針を作る
↓
タスクに分ける
↓
実装する
↓
検証する
```

この流れでは、Specは実装前の境界になります。

AIが実装に入る前に、人間が次を確認できます。

```text
目的は合っているか
範囲は広がっていないか
制約は守られているか
検証できる形になっているか
未決事項を勝手に決めていないか
```

## Spec設計の失敗パターン

### 1. 目的がない

何を解決するのかが曖昧だと、AIは見た目だけ整えたり、近そうな実装を選んだりします。

対策:

```text
誰の何の問題を解決するかを書く
```

### 2. 範囲がない

範囲が曖昧だと、AIは周辺の改善まで含めてしまうことがあります。

対策:

```text
In Scope / Out of Scopeを書く
```

### 3. 完了条件がない

完了条件がないと、どこまで実装すればよいか、人間もAIも判断しにくくなります。

対策:

```text
受け入れ条件を書く
例: 空入力時にエラーが出る、既存テストが通る、既存API仕様は変わらない
```

### 4. 検証方法が後付けになる

実装後に検証方法を考えると、確認漏れが起きやすくなります。

対策:

```text
Specの時点で検証方法を書く
```

### 5. 未決事項をAIが勝手に決める

未決のまま実装に入ると、AIがそれらしい判断で埋めてしまうことがあります。

対策:

```text
未決事項は未決として明示する
勝手に決めてよい範囲と、人間確認が必要な範囲を分ける
```

---

## まとめ

Spec設計は、AIに細かい命令を増やすことではありません。

目的は次です。

```text
AIが速く作れる状態ではなく、AIがズレずに作れる状態を作ること。
```

そのためには、次の考え方が重要です。

```text
目的を書く
範囲を書く
やらないことを書く
制約を書く
完了条件を書く
検証方法を書く
未決事項を分ける
必要に応じて契約仕様にする
```

AI時代のSpecは、重いドキュメントではなく、AI作業の境界線です。
