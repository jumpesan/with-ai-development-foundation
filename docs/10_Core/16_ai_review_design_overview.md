# AI時代のReview設計の整理

## この資料について

この資料は、withAI開発における **Review（レビュー）設計** の概要を共有するためのものです。

ここでいうReviewとは、AI Agentが作る計画・差分・Pull Requestを、人間が責任を持って採用判断できる状態に整えるための仕組みです。

Context、Spec、Issue / PR、Eval / Testとつなげると、位置づけは次のようになります。

```text
Context:
このProjectで常に守るルール

Spec:
今回の作業で何を達成するか

Issue:
AIに渡す作業チケット

Plan Review:
AIが実装に入る前に、方針・範囲・リスクを確認するレビュー

PR:
AIが作った変更を検査する変更提案

Eval / Test:
AI成果物が本当に条件を満たすか検査する仕組み

Review:
AIの計画や成果物を採用してよいか判断する仕組み
```

Reviewは、単に差分を読む作業ではありません。

withAI開発では、AIが作った変更をそのまま信じるのではなく、目的・範囲・設計・リスク・保守性を確認し、人間が責任を持って取り込める状態にする必要があります。

---

## Reviewがないと何が起きるか

AI成果物は、見た目や説明が自然でも、必ず正しいとは限りません。

テストが通っていても、次のような問題は残ります。

```text
Issueの目的とズレている
対象外の変更が混ざっている
既存の責務境界を壊している
将来の保守性が落ちている
重要なリスクを見落としている
PR説明と実際のdiffが一致していない
AIがもっともらしい説明をしているが、実際の変更と合っていない
```

Eval / Testは、AI成果物が条件を満たしているかを検査する仕組みです。

一方でReviewは、その計画や変更を採用してよいかを判断する仕組みです。

```text
Eval / Test:
壊れていないか、条件を満たすかを検査する

Review:
目的・設計・責務・保守性・リスクを見て、進めてよいか / 取り込んでよいか判断する
```

Review課題とは、AIの計画や成果物を人間が採用判断できるように整理し、確認し、最終判断するための課題です。

---

## Reviewで見るもの

Reviewでは、コードだけを見るのではなく、複数の情報を合わせて見ます。

```text
Issue
Spec
Plan
PR説明
diff
実行されたtest
CI結果
未確認事項
リスク
```

AIの説明だけを見ると、実際の差分とのズレに気づけない可能性があります。

そのため、Reviewでは次の対応を見る必要があります。

```text
Issueの目的とdiffが合っているか
Specの範囲を超えていないか
Planと実装が一致しているか
PR説明とdiffが一致しているか
検証結果が実際に残っているか
```

---

## Plan ReviewとPR Review

AI時代のReviewは、実装後だけではありません。

実装前にもReviewを置くと、手戻りを減らせます。

```text
Plan Review:
AIが実装に入る前に、方針・範囲・リスク・検証方法を見る

PR Review:
AIが作った差分を見て、採用してよいか判断する
```

Plan Reviewでは、まだ差分はありません。

そのため、見るべきものはコードではなく、方向性です。

```text
目的は合っているか
変更範囲は適切か
権限や外部影響が変わらないか
検証方法はあるか
未決事項をAIが勝手に決めていないか
```

PR Reviewでは、実際の成果物を見ます。

```text
diffは目的に合っているか
不要な変更が混ざっていないか
testは通っているか
PR説明は正しいか
人間が受け入れてよい変更か
```

---

## AI Reviewをどう扱うか

AIはReview補助にも使えます。

これは有効ですが、AI Reviewを最終判断者にしてはいけません。

AI Reviewには、次のような使い方が向いています。

```text
変更範囲の要約
潜在的な問題の指摘
テスト不足の候補出し
Context違反の検出補助
PR説明とdiffの不一致検出
```

一方で、次の判断は人間側に残す必要があります。

```text
この変更を採用してよいか
リスクを受け入れてよいか
設計方針として妥当か
将来の保守性をどう見るか
責任を持ってmergeできるか
```

AI Reviewは、人間の判断を置き換えるものではなく、人間が判断しやすくするための補助として扱う方が安全です。

---

## Reviewの失敗パターン

### 1. AIの説明をそのまま信じる

AIが自然な説明を書くと、正しそうに見えます。

しかし、説明とdiffが一致しているとは限りません。

対策:

```text
PR説明ではなくdiffを見る
Issue / Spec / Planとの対応を見る
```

### 2. テストが通ったのでReviewを省く

テストが通っていても、目的や設計とズレている可能性があります。

対策:

```text
Eval / TestとReviewを分ける
```

### 3. 差分が大きすぎる

差分が大きいと、人間が実質的にReviewできなくなります。

対策:

```text
IssueとPRを小さくする
Plan Reviewで範囲を絞る
```

### 4. Review観点が残っていない

毎回レビュー観点が変わると、品質が安定しません。

対策:

```text
Review checklistをContextやPR templateに残す
```

### 5. AI Reviewと人間Reviewの責任が混ざる

AIが指摘したから大丈夫、人間がApproveしたから大丈夫、という状態では責任が曖昧になります。

対策:

```text
AIは補助
人間が採用判断
という役割を明示する
```

---

## Reviewを支えるContext

Review観点は、毎回その場で思い出すより、Project側に残しておく方が安定します。

```text
PR template
review checklist
AGENTS.md
CONTRIBUTING.md
docs/review.md
```

たとえば、Contextに次のような観点があると、AI Agentも人間も同じ前提でReviewできます。

```text
generated filesは直接編集していないか
認証や権限に関係する変更は追加確認する
UI変更時はE2E確認を見る
DB schema変更時はmigrationを確認する
PR本文に検証結果を書く
```

Review設計は、Review担当者だけの問題ではありません。

Project側にReview観点を残すContext設計ともつながります。

---

## まとめ

AI時代のReview設計は、AIの成果物を疑うためだけのものではありません。

目的は次です。

```text
AIの計画や成果物を、人間が責任を持って採用判断できる状態にすること。
```

そのためには、次の考え方が重要です。

```text
AIの説明だけでなくdiffを見る
Issue / Spec / Planとの対応を見る
Eval / TestとReviewを分ける
Plan Reviewで作る前に確認する
PR Reviewで作った後に判断する
AI Reviewは補助として使う
最終判断は人間が持つ
Review観点をProject側に残す
```

Reviewは、AIの速度を止めるためではなく、AIの成果物を安全に採用するための判断基盤です。
