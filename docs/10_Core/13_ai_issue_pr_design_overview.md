# AI時代のIssue / PR設計の整理

## この資料について

この資料は、withAI開発における **Issue / PR設計** の概要を共有するためのものです。

ここでいうIssue / PR設計とは、AI Agentに作業を渡し、その成果物を人間と機械で確認できる単位に整えることです。

ContextやSpecとつなげると、位置づけは次のようになります。

```text
Context:
このProjectで常に守るルール

Spec:
今回の作業で何を達成するか

Issue:
AIに渡す作業チケット

PR:
AIが作った変更を検査する変更提案
```

Issue / PRは、従来も人間の開発管理やレビューで使われてきました。

withAI開発では、それに加えて **AIへの入力** と **AI成果物の出力・検査単位** という意味が強くなります。

---

## Issueとは

Issueは、作業チケットや課題票のようなものです。

人間だけの開発では、Issueは主に作業管理・担当者管理・進捗管理のために使われます。

withAI開発では、それに加えてAIに渡す作業指示として機能します。

```text
人間向けIssue:
作業管理・担当者管理・進捗管理の意味が強い

AI向けIssue:
作業指示・制約・完了条件・検証方法の意味が強い
```

AIに渡すIssueが曖昧だと、AIは不足している前提を自分で補います。

その結果、目的と違う作業、広すぎる変更、レビューしにくい差分が生まれやすくなります。

---

## PRとは

PRは **Pull Request（プルリクエスト）** の略です。

ざっくり言うと、次のような意味です。

```text
この変更を本流のbranchに取り込んでください
```

PRは、作業用branchで行った変更を、main branchなどの本流に取り込むための **変更提案** です。

PRには通常、次の情報が含まれます。

```text
何を変更したか
なぜ変更したか
どのファイルが変わったか
どのテストをしたか
レビューコメント
CI結果
```

withAI開発では、PRはAI成果物を検査するための重要な単位になります。

AIが作った差分をそのまま信じるのではなく、PR単位で確認できる状態にする必要があります。

---

## Issue / PRがないと何が起きるか

AI Agentに直接「これ直して」と渡すと、作業単位が曖昧になります。

```text
何を直すのか曖昧
どこまで触ってよいか曖昧
完了条件が曖昧
検証方法が曖昧
レビュー対象が曖昧
```

作業単位が曖昧なままAIが実装すると、次のような問題が起きやすくなります。

```text
Issueの目的と関係ない変更が混ざる
差分が大きくなりすぎる
テスト結果がPRに残らない
人間が何を見ればよいか分からない
後から責任範囲を追えない
```

Issue / PR設計は、AIに作業を渡す入口と、AI成果物を確認する出口を作るための設計です。

---

## AI向けIssueに書くとよい情報

AI向けIssueには、AIが迷わず作業できるように、次の情報を入れるとよいです。

```text
目的
背景
変更範囲
変更しない範囲
参照すべきContext
守る制約
完了条件
検証方法
注意点
```

例:

```text
目的:
検索結果が0件のときに、利用者へ分かりやすいメッセージを表示する。

変更範囲:
検索結果表示コンポーネントのみ。

変更しない範囲:
検索API、検索条件、DB schemaは変更しない。

完了条件:
0件時に指定文言が表示される。
既存の検索結果表示は変わらない。

検証:
pnpm test を実行する。
```

AI向けIssueでは、特に「変更しない範囲」が重要です。

AIは周辺の改善まで広げやすいため、Out of Scopeを明示すると作業を絞りやすくなります。

---

## AI向けPRに書くとよい情報

AIが作ったPRには、人間が採用判断できる情報を残す必要があります。

```text
何を変更したか
なぜ変更したか
Issueとの対応
変更範囲
触っていない範囲
実行した検証
未実行の検証
残った懸念
```

特に重要なのは、検証結果を曖昧にしないことです。

```text
NG:
テストしました。

OK:
pnpm test を実行し、成功しました。

OK:
pnpm test は未実行です。理由: ローカル環境に依存サービスがないため。
```

AIがPR本文を書く場合でも、検証の有無や未確認事項は人間が判断できる形で残す必要があります。

---

## Issue / PR設計の失敗パターン

### 1. Issueが大きすぎる

Issueが大きすぎると、AIの差分も大きくなり、人間がレビューしにくくなります。

対策:

```text
1 Issue = 1目的 に近づける
大きい作業は調査・設計・実装・検証に分ける
```

### 2. 完了条件がない

完了条件がないと、AIも人間も終わりを判断できません。

対策:

```text
受け入れ条件を書く
検証方法を書く
未確認なら未確認と書く
```

### 3. PR説明が差分と合っていない

AIがもっともらしいPR説明を書いても、実際のdiffと一致しないことがあります。

対策:

```text
PR説明とdiffを照合する
変更ファイル一覧を見る
Issueとの対応を見る
```

### 4. 検証結果が残らない

検証したかどうかがPRに残らないと、あとから判断できません。

対策:

```text
実行したcommand
成功/失敗
未実行の理由
をPRに残す
```

### 5. AIへの作業指示とレビュー単位が分かれていない

AIに広い作業を投げ、PRも広くなると、途中で止めにくくなります。

対策:

```text
Issueで作業を分ける
PRで差分を分ける
小さく確認できる単位にする
```

---

## まとめ

Issue / PR設計は、AIに作業を渡すための単なるチケット管理ではありません。

目的は次です。

```text
AIに渡す作業を小さく明確にし、AIが作った成果物を人間が確認できる単位にすること。
```

そのためには、次の考え方が重要です。

```text
Issueには目的・範囲・完了条件を書く
PRには変更内容・検証結果・未確認事項を書く
作業単位を大きくしすぎない
IssueとPRの対応を追えるようにする
AIの説明ではなくdiffと検証結果を見る
```

AI時代のIssue / PRは、作業管理だけでなく、AI作業の入力・出力・検査をつなぐ境界です。
