# withAI開発基盤 Core 目次・前書き

## この文書の目的

この文書は、withAI開発基盤を考えるための **Core課題** の入口です。

Core課題とは、AI coding agent（AIに開発作業を任せる仕組み）を開発現場に入れる場合に、最初に理解しておくべき基本課題です。

詳細は各Markdownを参照します。

---

## 注意書き

AI開発支援ツールやAI coding agentの状況は、非常に速く変化しています。

この資料は、2026年6月時点の公開情報・事例・整理内容をもとにした現時点での整理です。  
半年後には、ツール側の機能、ベストプラクティス、リスク、開発現場での使われ方が変わっている可能性があります。

## 11. Context

対象ファイル:

```text
11_ai_context_design_overview_ja_20260530.md
```

Contextは、AIに常に守らせる現場ルール・設計ルール・制約です。

例:

```text
このrepoではどのコマンドを使うか
どのディレクトリに何があるか
触ってはいけない場所はどこか
テストはどう実行するか
命名規則や設計方針は何か
```

AIは人間のように現場の空気を読めません。  
そのため、暗黙知をContextとして明示する必要があります。

---

## 12. Spec

対象ファイル:

```text
12_ai_spec_design_overview_ja_20260530.md
```

Specは、今回の作業で何を達成するかを定義します。

Contextが「常に守るルール」だとすると、Specは「今回の作業の目的・範囲・完了条件」です。

```text
何を作るか
なぜ作るか
どこまでやるか
どこから先はやらないか
何を満たしたら完了か
```

---

## 13. Issue / PR

対象ファイル:

```text
13_ai_issue_pr_design_overview_ja_20260530.md
```

Issue / PRは、AIに渡す作業単位と、AIが作った変更を提出・確認する単位を扱います。

GitHubに慣れていない人向けに言うと、Issueは作業依頼票、PRは変更提出票に近いです。

```text
Issue:
何を直すか、何を作るかを書いた作業単位

PR:
作った変更を取り込んでよいか確認する単位
```

注意点:

```text
PRはPull Requestの略です。
Plan ReviewのPlanとは別です。
```

---

## 14. Plan Review

対象ファイル:

```text
14_ai_plan_review_overview_ja_20260601.md
```

Plan Reviewは、AIに実装させる前に、AIが提示した実装方針を確認する工程です。

```text
AIがコード変更に入る前に、
その方針で進めてよいかを見る
```

Plan ReviewはPR Reviewとは違います。

```text
Plan Review:
作る前に見る

PR Review:
作った後に見る
```

AIで製造が速くなるほど、人間判断が後工程でボトルネックになりやすくなります。  
Plan Reviewは、後工程に集中しがちな人間判断の負荷を、実装前に小さく分散する工程です。

---

## 15. Eval / Test

対象ファイル:

```text
15_ai_eval_test_design_overview_ja_20260530.md
```

Eval / Testは、AIが作った成果物が条件を満たしているかを検査する工程です。

```text
コードが動くか
既存機能を壊していないか
仕様を満たしているか
テストは妥当か
AIの判断は評価できるか
```

AIが作ったコードは、それっぽく見えることがあります。  
それっぽいことと正しいことは同じではありません。

---

## 16. Review

対象ファイル:

```text
16_ai_review_design_overview_ja_20260530.md
```

Reviewは、AIが作った成果物を採用してよいか判断する工程です。

ここで扱うReviewは、主に実装後の確認です。

```text
PR全体を見る
差分を見る
AI reviewの指摘を見る
人間が採用可否を判断する
```

Plan Reviewとは違います。

```text
Plan Review:
AIが実装する前に見る

Review:
AIが実装した後に見る
```

---

## 17. 責務分担

対象ファイル:

```text
17_ai_responsibility_design_overview_ja_20260530.md
```

責務分担は、人間とAIがどこを担当し、誰が最終責任を持つかを扱います。

AIがコードを書けることと、AIに任せてよいことは同じではありません。

```text
AIが作業する
人間が確認する
人間が承認する
人間が責任を持つ
```

## 読む時の注意

### 1. 既存工程に当てはめすぎない

日本の従来工程と比較すると理解しやすい部分はあります。

ただし、withAI開発ではAIが実装案やコードを高速に生成するため、従来工程をそのまま当てはめると誤解します。

比較は理解の補助として使い、最終的にはwithAI開発として再定義する必要があります。

### 2. AIに任せるほど、人間判断が重要になる

AIで最初に速くなるのは、主に製造です。

しかし、次の判断は人間側に残ります。

```text
方針を決める
承認する
責任を持つ
レビューする
本番反映を判断する
障害時に説明する
```

AIが速くなるほど、人間判断がボトルネックになりやすくなります。

### 3. すべてを重くすればよいわけではない

Context、Spec、Plan Review、Eval、Reviewを厚くしすぎると、運用が重くなります。

Coreではまず構造を理解します。  
Cost / ROI / 運用負荷は、後で横断軸として見る方針です。
