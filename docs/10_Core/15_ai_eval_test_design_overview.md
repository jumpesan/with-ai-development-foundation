# AI時代のEval / Test設計の整理

## この資料について

この資料は、withAI開発における **Eval / Test（評価・試験）設計** の概要を共有するためのものです。

ここでいうEval / Testとは、AI Agentが作った成果物を、信じるのではなく、検査し、落とせるようにする仕組みです。

Context、Spec、Issue / PRとつなげると、位置づけは次のようになります。

```text
Context:
このProjectで常に守るルール

Spec:
今回の作業で何を達成するか

Issue:
AIに渡す作業チケット

PR:
AIが作った変更を検査する変更提案

Eval / Test:
AI成果物が本当に条件を満たすか検査する仕組み
```

重要なのは、Eval / Testは「AIを信用するための儀式」ではないということです。

```text
AIが作ったものを信じる
ではなく、
AIが作ったものを落とせる状態にする
```

これがwithAI開発におけるEval / Testの基本的な役割です。

---

## Eval / Testがないと何が起きるか

AIは自然な説明や、それらしいコードを書くことができます。

しかし、それが本当に正しいとは限りません。

```text
既存テストは通るが仕様とズレている
小さい関数は正しいが結合すると壊れる
UI上ではボタンが押せない
生成物の出力形式が少し変わっている
AIが追加したテストが弱い
AI reviewがもっともらしいが間違っている
```

AI成果物は、説明が自然でも、動作が正しいとは限りません。

そのため、withAI開発では次を考える必要があります。

```text
AI成果物をどう検査するか
既存テストで十分か
AIが作ったテストをどう信用するか
検査基準自体が正しいか
どの検査を自動化するか
人間は何を判断するか
```

Eval / Test課題とは、AI成果物を本当に検査できる状態にするための課題です。

---

## TestとEvalの違い

Testは、比較的明確な期待結果を確認する仕組みです。

```text
関数の戻り値が正しいか
APIのresponseが正しいか
画面に必要な要素が表示されるか
typecheckが通るか
buildが通るか
```

Evalは、より広い意味で、成果物が目的に合っているかを評価する仕組みです。

```text
回答品質
仕様との一致
安全性
使いやすさ
保守性
既存方針との整合
```

AI時代には、TestだけでなくEvalも重要になります。

なぜなら、AIは「テストは通るが、意図とは違うもの」を作ることがあるからです。

---

## Eval / Testで見ること

Eval / Testでは、AI成果物を複数の層で確認します。

```text
format
lint
typecheck
unit test
integration test
E2E test
snapshot test
security scan
manual review
```

機械的に確認できるものは、できるだけ自動化します。

```text
format違反
type error
unit test failure
生成物の直接編集
secret混入
lockfile更新漏れ
```

一方で、人間が判断すべきものも残ります。

```text
仕様に合っているか
使いやすいか
責務境界を壊していないか
保守できるか
リスクを受け入れてよいか
```

Eval / Test設計では、自動で落とすものと、人間が判断するものを分けることが重要です。

---

## AIが作ったテストをどう扱うか

AIはテストも作れます。

これは便利ですが、AIが作ったテストをそのまま信用しすぎると危険です。

AIが作ったテストには、次のような問題が混ざることがあります。

```text
実装に合わせただけのテスト
異常系が足りないテスト
仕様ではなく現在の挙動を固定するテスト
弱いassertion
重要な境界値がないテスト
```

AIにテストを書かせること自体は有効です。

ただし、テストの質を確認する観点が必要です。

```text
Specの受け入れ条件を確認しているか
正常系だけでなく異常系があるか
境界値があるか
失敗すべきケースが失敗するか
実装詳細に依存しすぎていないか
```

---

## Eval / Testの失敗パターン

### 1. テストが通っただけで安心する

テストが通っても、仕様とズレている可能性があります。

対策:

```text
Specの受け入れ条件と照合する
Issueの目的に戻って確認する
```

### 2. AIが作ったテストだけを見る

AIが自分に都合のよいテストを作る可能性があります。

対策:

```text
既存テスト、人間が用意した受け入れ条件、CIを組み合わせる
```

### 3. 検査対象と検査装置が近すぎる

AIが実装だけでなく、検査そのものも自由に変更できると危険です。

対策:

```text
重要な検査はrequired checkやbranch protectionで守る
成果物側から簡単に無効化できない場所に置く
```

### 4. 検証結果がPRに残らない

あとから何を確認したのか分からなくなります。

対策:

```text
実行したcommand
成功/失敗
未実行の理由
をPRに残す
```

### 5. 人間が見るべきものまで自動化しようとする

すべてを自動化しようとすると、判断すべき点が抜けます。

対策:

```text
機械が見るものと人間が判断するものを分ける
```

---

## Eval / TestとContext

Eval / TestはContextとつながります。

Contextに検証の入口が書かれていなければ、AI Agentは何を実行すべきか分かりません。

```text
UI変更時はE2E testを見る
API変更時はcontract testを見る
生成物変更時はgenerator sourceを見る
schema変更時はmigration testを見る
```

このような情報をProject側のContextに残しておくと、AI Agentも人間も同じ検証方針に戻れます。

---

## まとめ

Eval / Test設計は、AI成果物を信じるためのものではありません。

目的は次です。

```text
AI成果物を検査し、条件を満たさないものを落とせる状態にすること。
```

そのためには、次の考え方が重要です。

```text
Specの受け入れ条件とつなげる
Issue / PRに検証結果を残す
機械で落とせるものは自動で落とす
人間が判断すべきものを残す
AIが作ったテストもレビューする
検査装置を成果物側から守る
```

AI時代のEval / Testは、AIを信用するためではなく、AIを安全に使うための検査基盤です。
