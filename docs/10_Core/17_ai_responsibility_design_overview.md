# AI時代の責務分担設計の整理

## この資料について

この資料は、withAI開発における **責務分担設計** の概要を共有するためのものです。

ここでいう責務分担とは、AI Agentと人間が、どこを担当し、どこで人間が判断し、誰が最終責任を持つかを決めることです。

Context、Spec、Issue / PR、Eval / Test、Reviewとつなげると、位置づけは次のようになります。

```text
Context:
このProjectで常に守るルール

Spec:
今回の作業で何を達成するか

Issue:
AIに渡す作業チケット

Eval / Test:
AI成果物が本当に条件を満たすか検査する仕組み

Review:
AIの計画や成果物を採用してよいか判断する仕組み

責務分担:
人間とAIがどこを担当し、誰が責任を持つかを決める仕組み
```

AIは、調査・実装・テスト案作成・レビュー補助など、開発の多くの場面で使えます。

しかし、AIが作業できることと、AIに任せてよいことは同じではありません。

```text
AIができる
=
AIに任せてよい
ではない
```

withAI開発では、AIにどこまで権限を渡すか、人間がどこで介入するか、誰が最終責任を持つかを明確にする必要があります。

---

## 責務分担がないと何が起きるか

AIを開発に入れると、次のような状態が起きます。

```text
AIが実装した
AIがテストを書いた
AIがレビューコメントを出した
CIは通った
人間がApproveした
```

この時、重要なのは次です。

```text
誰が目的を決めたのか
誰が設計方針を判断したのか
誰がリスクを受け入れたのか
誰がmergeしてよいと判断したのか
問題が起きた時に誰が説明責任を持つのか
```

ここが曖昧だと、AIを使った結果だけが残り、判断と責任が見えなくなります。

責務分担設計は、AIを作業者として含めた時に、人間とAIの担当範囲・承認・責任・追跡をどう設計するかという課題です。

---

## AIに任せやすいこと

AIには、任せやすい作業があります。

```text
調査のたたき台
コード修正案の作成
テストケース案の作成
ドキュメント草案
差分の要約
レビュー観点の候補出し
単純な変換作業
```

これらは、AIが速く処理しやすく、人間があとから確認しやすい作業です。

ただし、任せやすい作業でも、最終確認は必要です。

```text
AIが作った
↓
人間や機械が確認する
↓
採用するか判断する
```

---

## AIに任せすぎると危険なこと

AIに任せる範囲が広すぎると、危険な領域があります。

```text
本番環境への操作
secretやcredentialの扱い
権限変更
外部サービスへの書き込み
契約・法務・コンプライアンス判断
Security riskの受け入れ判断
merge / releaseの最終判断
```

これらは、AIが補助することはあっても、AIだけで完結させるべきではありません。

特に、次のような判断は人間側に残す必要があります。

```text
目的を決める
リスクを受け入れる
例外を承認する
本番に反映する
最終的な説明責任を持つ
```

---

## 人間が持つべき責務

AIを使っても、人間側に残る責務があります。

```text
何を作るかを決める
なぜ作るかを説明する
AIに渡す範囲を決める
危険な操作を承認する
AI成果物を採用してよいか判断する
問題が起きた時に説明する
```

AIは作業を速くできます。

しかし、目的、価値判断、リスク受容、責任説明は人間の側に残ります。

これは、AIを信用しないという意味ではありません。

AIを安全に使うために、AIに任せる部分と人間が判断する部分を分けるという意味です。

---

## 責務分担を明示する方法

責務分担は、口頭や暗黙の前提だけに置かない方が安全です。

Project側に残すと、AI Agentも人間も同じ前提で作業できます。

```text
AGENTS.md
PR template
review checklist
approval rule
decision_inbox.md
release手順
incident対応手順
```

たとえば、次のように明示できます。

```text
AI Agent:
実装案を作る
テスト案を作る
PR説明の草案を書く

人間:
目的を決める
Planを承認する
PRを採用するか判断する
Security riskを受け入れるか判断する
```

このように分けると、AIに任せる作業と、人間が責任を持つ判断を混同しにくくなります。

---

## 承認と責任

承認は、単に「OK」を押すことではありません。

人間が判断できる情報が揃っていなければ、承認は形だけになります。

承認時には、次のような情報が必要です。

```text
何をしようとしているか
どの範囲に影響するか
どの権限を使うか
どの検証をしたか
どのリスクが残っているか
rollbackできるか
```

責務分担設計では、人間が責任を持って承認できる状態を作る必要があります。

AIが作業したから人間の責任が消えるわけではありません。

AIを使うほど、人間が判断するための情報を揃えることが重要になります。

---

## 責務分担の失敗パターン

### 1. AIに任せたから大丈夫だと思う

AIが自然な説明をしても、正しいとは限りません。

対策:

```text
AI成果物をEval / Test / Reviewに通す
```

### 2. 人間承認が形だけになる

承認者が判断できる情報を持っていないと、承認は責任の押し付けになります。

対策:

```text
承認に必要な情報をPlanやPRに残す
```

### 3. 最終責任者が曖昧になる

AI、人間、CI、レビュー担当のどこに責任があるのか分からなくなります。

対策:

```text
誰が採用判断をしたか
誰がリスクを受け入れたか
を記録する
```

### 4. AIに危険操作まで任せる

実装と本番操作、提案と承認が混ざると危険です。

対策:

```text
作業権限と承認権限を分ける
危険操作は人間確認を必須にする
```

### 5. AI Reviewを最終判断にしてしまう

AI Reviewは補助として有効ですが、最終判断者ではありません。

対策:

```text
AI Review = 補助
人間 Review = 採用判断
と分ける
```

## 短命Agentと責務分担

短命Agentを前提にすると、責務分担はより明示しやすくなります。

Agentは作業ごとに入れ替わるため、継続すべき判断や責任はAgentの記憶ではなくProject側に残す必要があります。

```text
Agent:
作業する
提案する
要約する
検証結果を書く

Project側:
判断を残す
承認を残す
例外理由を残す
責任範囲を残す
```

この考え方では、AI Agentは作業者であり、Project側のContextと人間の判断が継続性を持ちます。

---

## まとめ

責務分担設計は、AIに何を任せるかを決めるだけではありません。

目的は次です。

```text
AIを作業者として使いながら、人間が判断と説明責任を持てる状態を作ること。
```

そのためには、次の考え方が重要です。

```text
AIができることと、任せてよいことを分ける
AIの作業と人間の判断を分ける
作業権限と承認権限を分ける
危険操作には人間確認を置く
承認できる情報を揃える
判断と例外理由をProject側に残す
最終責任は人間が持つ
```

AIを使うことは、責任をAIに移すことではありません。

AIを安全に使うために、責任を見える形にすることが重要です。
