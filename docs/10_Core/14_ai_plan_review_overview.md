# AI時代のPlan Review整理

## この資料について

この資料は、withAI開発における **Plan Review** の位置づけを整理するためのものです。

Plan Reviewとは、AI Agentに実装させる前に、AIが提示した実装方針・変更範囲・リスク・検証方法を人間が確認することです。

AIは実装速度が速いため、方向がズレたまま進むと、短時間で大きな差分が生まれます。

Plan Reviewは、その前に一度立ち止まるための制御点です。

---

## Plan ReviewとPR Reviewの違い

Plan ReviewとPR Reviewは、見るタイミングが違います。

```text
Plan Review:
AIがコード変更に入る前に、実装方針を見る

PR Review:
コード変更後に、Pull Requestの差分を見る
```

PR Reviewは「もう差分が作られた後」の確認です。

一方、Plan Reviewは「差分を作る前」の確認です。

```text
Plan Review:
作る前に止める

Diff Review:
作った後に差分を見る

PR Review:
提出された変更を採用してよいか見る
```

Plan Reviewの本質は、AIが間違った方向へ高速に進む前に、人間が方向を確認することです。

---

## Plan Reviewがないと何が起きるか

AIに曖昧な依頼を渡すと、AIはそれらしい実装方針を作り、そのまま差分を作れます。

```text
曖昧な依頼
↓
AIがそれっぽく解釈する
↓
AIが実装する
↓
差分が大きくなる
↓
人間が後から確認する
↓
レビュー不能・手戻り・責任不明確化が起きる
```

この流れでは、人間は最後に大きな差分を渡されます。

その時点で方向が違っていると、修正よりもやり直しに近くなります。

Plan Reviewは、実装前に次を確認するための場です。

```text
目的は合っているか
変更範囲は広がっていないか
設計方針は妥当か
リスクを見落としていないか
検証方法はあるか
未決事項をAIが勝手に決めていないか
```

---

## Plan Reviewの位置づけ

withAI開発の流れに置くと、Plan Reviewは次の位置にあります。

```text
Context
↓
Spec
↓
Issue
↓
Plan Review
↓
Implementation
↓
Eval / Test
↓
PR Review
↓
Merge
```

Plan Reviewは、Specと実装の間に置く確認です。

Specで「何を作るか」を決め、Plan Reviewで「どう作ろうとしているか」を確認します。

---

## Plan Reviewで見ること

Plan Reviewでは、コードそのものではなく、実装に入る前の計画を見ます。

見る観点は次のようなものです。

```text
Issue / Specとの対応
変更範囲
変更しない範囲
設計方針
影響範囲
リスク
検証方法
人間判断が必要な点
```

特に重要なのは、AIがどこまで自分で判断しようとしているかです。

```text
AIが決めてよいこと
AIが提案してよいが、人間が確認すべきこと
AIが勝手に決めてはいけないこと
```

この境界を確認することで、AIに任せる作業を安全に絞り込めます。

---

## Plan Reviewで使いやすい確認項目

```text
このPlanはIssueの目的に合っているか
Specにない作業が混ざっていないか
Out of Scopeを守っているか
変更対象fileは妥当か
既存設計を壊していないか
権限や外部影響が変わらないか
生成物を直接編集しようとしていないか
検証commandが具体的か
失敗時の戻し方が見えているか
```

Plan Reviewでは、細かい実装行を読むより、作業の方向と境界を見ることが重要です。

---

## Plan Reviewの失敗パターン

### 1. Planが抽象的すぎる

「適切に修正する」「既存に合わせる」だけでは、何をするのか分かりません。

対策:

```text
変更対象
変更内容
検証方法
を具体化する
```

### 2. 変更範囲が広すぎる

AIが周辺改善まで含めると、差分が大きくなります。

対策:

```text
In Scope / Out of Scopeを確認する
```

### 3. 検証方法がない

実装後に何で確認するかがないPlanは危険です。

対策:

```text
Planの時点でtest / typecheck / buildなどの確認方法を書く
```

### 4. 未決事項をAIが埋めている

仕様にない判断をAIが勝手に決めると、後で方針とズレます。

対策:

```text
未決事項を明示し、人間確認へ戻す
```

### 5. Plan Reviewが承認儀式になる

人間が判断できる情報がないまま「OK」すると、Plan Reviewは形だけになります。

対策:

```text
承認するために必要な情報をPlanに含める
```

## Plan Reviewと短命Agent

短命Agentを前提にすると、Plan Reviewはより重要になります。

作業ごとにAgentが変わる場合、前回の会話や暗黙の前提に頼れません。

そのため、Planに次を残しておくと、別のAgentや人間が後から見ても判断しやすくなります。

```text
何をしようとしていたか
なぜその方針にしたか
どこを触る予定だったか
どこは触らない予定だったか
何で検証する予定だったか
```

Plan Reviewは、AIに作業させる前の確認であると同時に、作業意図をProject側に残す役割も持ちます。

---

## まとめ

Plan Reviewは、AIに作業させる前に方向を確認するための仕組みです。

目的は次です。

```text
AIが間違った方向へ高速に進む前に止めること。
```

そのためには、次の考え方が重要です。

```text
コードを書く前にPlanを見る
Issue / Specとの対応を見る
変更範囲を見る
リスクを見る
検証方法を見る
未決事項を人間に戻す
承認できる情報を揃える
```

Plan Reviewは、AIの速度を落とすためのものではありません。

AIの速度を活かすために、間違った方向へ進まないようにするための制御点です。
