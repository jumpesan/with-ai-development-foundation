# withAI開発の全体像

## これは何か

最近のAI開発支援は、単に「ChatGPTに聞く」「GitHub Copilotに補完してもらう」という段階から、かなり大きく変わってきています。

今は、AIがコードを書く補助をするだけでなく、リポジトリを読み、ファイルを編集し、テストを実行し、PRを作り、レビューにも参加する方向へ進化しています。

つまり、開発の中心テーマは次のように変わりつつあります。

```text
AIを使って開発する
↓
AIが安全に参加できる開発システムを設計する
```

この文書は、今のwithAI開発の全体像をざっくり把握するためのものです。

---

## 全体像

withAI開発は、次の8つの層で見ると分かりやすいです。

```text
1. Context     : AIに何を読ませるか
2. Spec/Task   : AIに何を頼むか
3. Agent       : AIがどう作業するか
4. Review/Test : AI成果物をどう検査するか
5. Safety      : AI作業をどう安全にするか
6. Tooling     : AIにどの道具を渡すか
7. Observe     : AI作業をどう観測するか
8. Learn       : 人間と組織がどう進化するか
```

---

## 1. Context: AIに何を読ませるか

AIはコードだけを読んでも、プロジェクトの意図やルールまでは十分に分かりません。

たとえば、次のような情報がないと、もっともらしいけれどズレた修正をする可能性があります。

```text
- このプロジェクトの目的
- ディレクトリ構成
- build方法
- test方法
- lint/typecheck方法
- 触ってよいファイル
- 触ってはいけないファイル
- 生成物の扱い
- セキュリティ上の注意
- コーディング規約
- 過去の設計判断
```

そのため最近は、AI向けに次のようなファイルを置く動きが出ています。

```text
- AGENTS.md
- CLAUDE.md
- copilot-instructions.md
- Cursor rules
```

これらは、人間向けのREADMEとは少し違い、AIに作業させるための文脈ファイルです。

重要なのは、AIに「コード」だけでなく「作業文脈」を渡すことです。

ただし、こうしたファイルを書けばよいという話ではありません。長すぎる、古い、実行できない、禁止事項ばかり、といった状態になると逆に邪魔になります。

重要なのは、AIが作業するために必要な最小限で正確な文脈を保つことです。

---

## 2. Spec / Task: AIに何を頼むか

AIに「いい感じに直して」と頼むと、速く作業してくれる一方で、意図と違うものができる可能性も高くなります。

そのため、AIに渡す作業単位が重要になります。

```text
- 何を直すのか
- なぜ直すのか
- 完了条件は何か
- 触ってよい範囲はどこか
- 触らない範囲はどこか
- どのテストを通すべきか
- どのようにレビューすべきか
```

これからの開発では、IssueやPRの意味が少し変わってきます。

```text
Issue = AIへの作業仕様書
PR    = AI成果物の検査単位
```

AIに任せるほど、作業を小さく、具体的に、検証可能に分解する力が重要になります。

今後強いのは、単にプロンプトがうまい人ではなく、AIが失敗しにくい作業単位を設計できる人です。

---

## 3. Agent: AIがどう作業するか

AIは、すでにチャットで相談に乗るだけの存在ではなくなりつつあります。

今の開発向けAIエージェントは、次のようなことを行う方向に進んでいます。

```text
- リポジトリを読む
- ファイルを編集する
- コマンドを実行する
- テストを回す
- エラーを見て修正する
- branchを作る
- PRを作る
- レビューコメントに対応する
```

代表的なものとして、次のようなツールがあります。

```text
- ChatGPT / Codex
- GitHub Copilot coding agent
- Claude Code
- Aider
- OpenHands
- Continue
- Cline
- Cursor
```

ここでの大きな変化は、AIが「助言者」から「権限を持つ作業者」に近づいていることです。

```text
AI = 助言者
↓
AI = 権限を持つ作業者
```

そのため、便利さだけでなく、権限、安全、検証、責任が重要になります。

---

## 4. Review / Test: AI成果物をどう検査するか

AIがコードを書くほど、人間レビューだけでは追いつきにくくなります。

AIは速く変更できますが、その変更が正しいとは限りません。

そのため、withAI開発では検証基盤が非常に重要になります。

```text
- unit test
- integration test
- E2E test
- snapshot test
- golden file test
- lint
- typecheck
- benchmark
- AI review
- PR check
- LLM eval
```

特に重要なのは、既存のテストを通すだけで安心しないことです。

AIは、次のような失敗をすることがあります。

```text
- 既存テストは通るが仕様とズレている
- 局所修正は正しいが全体設計を壊す
- 生成物を直接編集してしまう
- 一見自然なコードだが保守性が落ちる
- セキュリティ上危ない実装を入れる
- 依存関係を安易に追加する
```

withAI開発で差がつくのは、実装速度よりも検証設計です。

```text
AIに作らせるなら、検査基盤を先に強くする
```

テストが弱い状態でAIを使うと、速く作れるのではなく、速く壊せる状態になりかねません。

---

## 5. Safety: AI作業をどう安全にするか

AIがterminal、file、network、browser、外部toolを扱えるようになると、安全性の重要度が一気に上がります。

考えるべきリスクには、次のようなものがあります。

```text
- secret漏洩
- 危険コマンド実行
- 不要な外部通信
- ファイル削除
- 意図しない依存追加
- 悪意あるpackageの導入
- prompt injection
- sandbox外への情報流出
- 権限過多
```

そのため、AIに作業させる場合は、次のような設計が必要になります。

```text
- 最小権限
- sandbox
- devcontainer
- git worktree
- secret scanning
- 依存関係チェック
- network制御
- 実行前承認
- 危険操作のdeny rule
- CIでの検証
```

重要なのは、禁止を増やすだけではないことです。

禁止ばかりにすると、AIの実行力が落ちます。  
一方で、自由にしすぎると危険です。

理想は、リスクに応じて操作を分けることです。

```text
低リスク操作   : 自動許可
中リスク操作   : 確認付きで許可
高リスク操作   : 原則禁止または人間承認
危険操作       : 明示的に拒否
```

---

## 6. Tooling: AIにどの道具を渡すか

これからのAIは、単に会話するだけではなく、さまざまなtoolを使うようになります。

```text
- GitHub
- issue tracker
- browser
- database
- filesystem
- test runner
- documentation
- API
- CI/CD
- monitoring
```

最近は、MCPのように、AIと外部toolをつなぐための標準化された仕組みも出てきています。

ここでの本質は、AIに道具を渡す時代になっていることです。

```text
AIに道具を渡す
↓
AIが開発環境の一部として動く
```

ただし、toolを渡すほど安全性・権限・ログ・再現性が重要になります。

つまり、ToolingはSafetyとセットで考える必要があります。

---

## 7. Observe: AI作業をどう観測するか

AIの作業はブラックボックスになりやすいです。

何を読んだのか、なぜその修正をしたのか、どのテストを実行したのか、どこで失敗したのかが見えないと、改善できません。

観測したいものは次のような情報です。

```text
- AIが読んだファイル
- AIが変更したファイル
- 実行したコマンド
- 実行したテスト
- 失敗した理由
- 修正の判断理由
- PRでの差分
- review結果
- CI結果
- token/cost
- 作業時間
```

ここでの本質は、次です。

```text
観測できないAI開発は改善できない
```

AIに作業させるなら、会話ログだけでなく、検証可能な作業証跡を残すことが重要になります。

---

## 8. Learn: 人間と組織がどう進化するか

withAI開発では、人間の役割が消えるわけではありません。  
むしろ、人間の役割が変わります。

これから強い開発者は、単にコードを書く人ではなく、次のような力を持つ人です。

```text
- 要件を構造化できる
- 仕様を明確にできる
- AIに渡す作業単位を作れる
- AIの成果物をレビューできる
- テスト・検証観点を設計できる
- 失敗パターンをルール化できる
- 文脈を資産化できる
- 安全にAIを運用できる
```

AI時代に重要になるのは、次のような設計力です。

```text
Context設計
Task設計
Review設計
Test設計
Safety設計
Learning loop設計
```

AIを使えば楽になる部分は増えます。  
ただし、AIに任せすぎると、理解が浅くなったり、判断力が落ちたりする可能性もあります。

大事なのは、AIを自分の代わりにすることではなく、自分の判断力と開発力を拡張することです。

---

## 今起きている変化

少し前までは、AI開発支援はこのようなものでした。

```text
ChatGPTに聞く
Copilotに補完してもらう
```

今は、こう変わりつつあります。

```text
AIにrepo文脈を渡す
AIにIssueを割り当てる
AIがbranchを切る
AIがtestを回す
AIがPRを出す
AIがPRをレビューする
人間が設計・判断・責任を持つ
```

つまり、AIは単なる補助ツールではなく、開発プロセスに参加する存在になりつつあります。

---

## 重要な考え方

withAI開発で重要なのは、流行りのツールを追うことだけではありません。

大事なのは、各ツールを次のように見ることです。

```text
このツールは、自分たちの開発基盤のどの層を強くするのか？
```

たとえば、

```text
AGENTS.md      → Contextを強くする
Issue template → Task設計を強くする
AI review      → Reviewを強くする
SWE-bench/eval → Test/Evalを強くする
sandbox        → Safetyを強くする
MCP            → Toolingを強くする
agent logs     → Observeを強くする
```

ツール名ではなく、どの課題に効くのかで見ると判断しやすくなります。

---

## まとめ

withAI開発の本質は、AIツールの導入ではありません。

```text
AIが安全に参加できる開発基盤の再設計
```

です。

その基盤は、次の要素でできています。

```text
- 文脈
- 作業単位
- agent運用
- 検証
- review
- safety
- tooling
- 観測
- 人間の判断
- 学習ループ
```

今後は、AIを少し便利に使う人と、AIが参加できる開発システムを設計する人で差が出てくる可能性があります。

withAI開発を本気で扱うなら、単にAIにコードを書かせるのではなく、AIが安全に、継続的に、検証可能な形で開発に参加できる仕組みを作ることが重要です。
